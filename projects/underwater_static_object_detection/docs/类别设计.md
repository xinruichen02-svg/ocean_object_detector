# 类别设计：水下静态物体识别

## 为什么先设计类别

目标检测不是“看到什么都识别”，而是先定义模型需要识别哪些类别。类别定义不清楚，后面会遇到这些问题：

- 不同数据集的类别名称对不上。
- 同一类目标在不同数据集中标注粒度不同。
- 模型训练后不知道应该服务哪种机器人任务。
- 评估时指标没有实际意义。

## 当前正式类别

第一阶段确定采用 DUO 的四个标准类别。YOLO 转换后的项目编号固定为：

```text
0: holothurian  # 海参
1: echinus      # 海胆
2: scallop      # 扇贝
3: starfish     # 海星
```

原因：

- 这四类就是 DUO 的目标类别，也常见于 UDD/UODD、URPC/UTDAC 系列。
- 多数是静态或低速底栖目标，适合水下机器人近底作业。
- DUO 提供目标检测边界框，能够保留目标类别和空间位置。

注意：以上是本项目转换后的连续编号。DUO 原始 COCO 标注中的 `category_id`
必须从实际 JSON 的 `categories` 字段读取后再映射，不能直接假设也是 0 到 3。

## 候选扩展类别

第二阶段再考虑：

```text
shell
conch
abalone
barnacle
reef
coral
stone
rock
seaweed
```

这些类别目前不建议一开始全部加入，因为：

- `shell`、`conch`、`abalone` 在部分数据集有，但公开检测数据不一定多。
- `reef`、`coral` 更像结构或区域，可能更适合分割。
- `stone`、`rock` 在水下公开数据中常是底质分类，不一定有检测框。
- `seaweed` 形状复杂，检测框可能不如分割合适。

## 石头类的当前判断

“石头类”对机器人避障和底质理解有价值，但它和海参、扇贝这类实例目标不同。

建议先把石头类放入候选方向：

```text
阶段 A：作为背景和失败案例记录。
阶段 B：用底质分类数据理解 stone/sand 场景。
阶段 C：如果确实需要机器人避障，再单独标注 rock/stone 检测或分割数据。
```

不要一开始就把 `stone` 当成和 `holothurian` 同等成熟的检测类别。
